# Data catalog

The structured policy catalog is stored in SQLite so it can support editorial
work now and a website-backed data model later.

## Why SQLite

- portable and easy to inspect
- strong enough for structured querying
- simple to migrate into a future web application
- good fit for preserving source provenance and imported IDs

## Database files

| File | Purpose |
|------|---------|
| `policy/catalog/policy_catalog_v2.sqlite` | Active catalog in the v2 ID format |
| `policy/catalog/policy_catalog.sqlite` | Legacy v1 catalog retained for provenance and audit |

## Current schema (`policy_catalog_v2.sqlite`)

### v2 ID format

Policy position IDs follow the pattern `XXXX-XXXX-0000` (regex:
`^[A-Z]{4}-[A-Z]{4}-[0-9]{4}$`), e.g. `HLTH-COVR-0001`. See
`policy/catalog/policy-id-schema.md` for the full specification.

### Tables

- `domains` — policy domains (`code`, `name`, `pillar_id`, `html_file`, `is_cross_domain`)
- `subdomains` — policy families within each domain (`code`, `domain`, `name`)
- `positions` — canonical policy stances (`id`, `domain`, `subdomain`, `seq`, `short_title`, `full_statement`, `plain_language`, `is_cross_domain`, `status`, `rule_notes`)
- `position_pillar_appearances` — where each position appears in pillar pages
- `legacy_id_map` — old-to-new ID provenance (`old_id`, `new_id`, `source`, `notes`)

### `positions` status values

| Status | Meaning |
|--------|---------|
| `CANONICAL` | Adopted, active policy position |
| `PROPOSED` | Under consideration |
| `DEPRECATED` | Superseded or withdrawn |
| `REVIEW` | Flagged for human review |

## Reconciliation status

Catalog and pillar HTML are maintained in sync during the current phase.

- Any new position added to HTML must be backfilled into the DB in the same commit.
- Any new position added to the DB should include a corresponding HTML card in the same commit.

## Build behavior

`policy_catalog_v2.sqlite` is generated by `scripts/build-catalog-v2.py`.
Re-running the script drops and recreates the DB from source data, and the
script is idempotent. Do not hand-edit the DB directly.

```bash
python3 scripts/build-catalog-v2.py \
  --output policy/catalog/policy_catalog_v2.sqlite \
  --report policy/catalog/migration-report.md
```

## Reports

Catalog reports live under `policy/catalog/`, including:

- `migration-report.md` — HTML and DB reconciliation report
- `citation-audit.md` — citation audit data for pillar pages
