# Data catalog

The structured policy catalog is stored in SQLite so it can support both editorial work now and a website-backed data model later.

## Why SQLite

- portable and easy to inspect
- strong enough for structured querying
- simple to migrate into a future web application
- good fit for preserving source provenance and imported IDs

## Database files

| File | Purpose |
|------|---------|
| `data/policy_catalog_v2.sqlite` | **Active catalog** — 3,810 positions in the v2 ID format |
| `data/policy_catalog.sqlite` | **Provenance only** — the original v1 catalog; retained for audit history; not read by any active script |

## Current schema (`policy_catalog_v2.sqlite`)

### v2 ID format

Policy position IDs follow the pattern `XXXX-XXXX-0000` (regex: `^[A-Z]{4}-[A-Z]{4}-[0-9]{4}$`), e.g. `HLTH-COVR-0001`. See `data/policy-id-schema.md` for the full specification.

### Tables

- `domains` — one per policy pillar (4-char code, name, `pillar_id`, `html_file`, `is_cross_domain`). `XDOM` is reserved for future cross-domain positions and has no HTML file.
- `subdomains` — 506 rows; policy families within a domain (`code` + `domain` FK, `name`). The same 4-char code may appear in multiple domains.
- `positions` — 3,810 rows; one per canonical policy stance (`id`, `domain`, `subdomain`, `seq`, `short_title`, `full_statement`, `plain_language`, `is_cross_domain`, `status`)
- `position_pillar_appearances` — maps each position to the pillar page(s) where it appears (`position_id`, `pillar_domain`, `section_name`, `display_order`)
- `legacy_id_map` — 3,810 rows; tracks old-to-new ID relationships (`old_id`, `new_id`, `source` \[`db`/`html`/`both`\], `notes`)

### `positions` status values

Each `positions` row carries a `status` field. Valid values:

| Status | Meaning |
|--------|---------|
| `CANONICAL` | Adopted, active policy position — the default for all migrated records |
| `PROPOSED` | Under consideration; not yet adopted |
| `DEPRECATED` | Superseded or withdrawn; retained for provenance only |
| `REVIEW` | Flagged during migration for human review — may have ambiguous domain mapping or other anomaly |

### `plain_language` field

The `plain_language` column on `positions` holds the 1–3 sentence plain-language summary for each position (~8th grade reading level). This column is fully populated as of April 2026 (all 3,810 positions).

## Reconciliation status

HTML ↔ DB reconciliation is complete as of April 2026. Gap = 0 in both directions.

- Any new position added to HTML must be backfilled into the DB in the same commit.
- Any new position added to the DB should include a corresponding HTML card in the same commit.

## Build behavior

`policy_catalog_v2.sqlite` is generated by `scripts/build-catalog-v2.py`. Re-running the script drops and recreates the DB from source data — it is idempotent. Do not hand-edit the DB directly; make changes to source data and regenerate.

```bash
python3 scripts/build-catalog-v2.py --output data/policy_catalog_v2.sqlite --report data/migration-report.md
```

Provenance for every position is preserved in `legacy_id_map`: each row records the `old_id`, `new_id`, and `source` (`db`, `html`, or `both`).

## Research reports (`data/pillar_reports/`)

This directory contains per-pillar policy audit reports, scope analysis CSVs,
and reconciliation reports. It does not contain PolicyOS documents — those live
in `policy/policyos/` at the repo root.

- `citation-audit.md` — full citation audit across all 25 pillar HTML files
- `migration-report.md` — HTML ↔ DB reconciliation report (most recent build)
- `human_report.md` — human-readable reconciliation summary

The v2 catalog was built by migrating and deduplicating positions from the v1 DB and all tagged HTML policy cards. A full reconciliation audit (HTML ↔ DB) is in progress. Until it completes:

- HTML additions must be backfilled into the DB in the same commit.
- DB additions should include a corresponding HTML card in the same commit.
- Treat both sources as valid; flag divergences for human review rather than auto-resolving them.
